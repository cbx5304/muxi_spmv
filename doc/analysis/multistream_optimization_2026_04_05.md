# 多流优化分析报告

## 测试日期: 2026-04-05

---

## 一、背景

在之前的测试中，我们发现Mars X201的SpMV性能受限于L2 Cache大小。本报告探索多流并行是否能进一步提升性能。

---

## 二、多流优化原理

### 2.1 为什么多流可能有效？

```
传统单流执行:
┌─────────────────────────────────────────────────────┐
│ H2D Transfer │ Kernel Exec │ D2H Transfer          │
└─────────────────────────────────────────────────────┘

多流执行 (理想情况):
┌─────────────────────────────────────────────────────┐
│ Stream 0: H2D │ Kernel 0 │ D2H                     │
│ Stream 1:     │ H2D │ Kernel 1 │ D2H               │
└─────────────────────────────────────────────────────┘
```

多流可以：
1. **提高SM利用率**: 不同kernel在不同SM上并行执行
2. **隐藏内存延迟**: 计算和传输重叠
3. **减少调度开销**: 更细粒度的任务分配

### 2.2 Mars X201 vs RTX 4090 架构差异

| 参数 | Mars X201 | RTX 4090 | 影响 |
|------|-----------|----------|------|
| SM数量 | 104 | 128 | Mars可能需要更多并行度 |
| Warp大小 | 64 | 32 | Mars单kernel利用率低 |
| L2 Cache | ~4MB | 72MB | Mars内存压力大 |

## 五、分析

### 5.1 为什么Mars X201受益于多流？

1. **SM利用率提升**
   - Mars X201有104个SM
   - 单流kernel可能无法充分利用所有SM
   - 多流允许不同kernel在不同SM上并行执行

2. **Warp效率问题**
   - Warp=64，单kernel每行最多64线程
   - avgNnz=10.7时，线程利用率低
   - 多流允许更多warp并行执行

3. **内存访问重叠**
   - 不同流的内存访问可以重叠
   - 减少PCIe传输延迟

### 5.2 为什么RTX 4090不受益？

1. **L2缓存足够大**
   - 72MB L2缓存可容纳x向量访问
   - 内存压力小，无需多流

2. **单流已高效**
   - 128个SM，单流kernel已能充分利用
   - 多流反而增加调度开销

3. **Warp效率高**
   - Warp=32，线程分配更灵活
   - 单kernel即可达到高利用率

### 5.3 为什么2流最优？

1. **数据传输瓶颈**
   - 77%+时间是数据传输
   - 过多流无法隐藏传输延迟

2. **SM数量匹配**
   - 104 SM / 2流 = 52 SM/流
   - 足够容纳单kernel的所有block

3. **调度开销**
   - 更多流增加调度开销
   - 但收益递减

## 六、结论

### 6.1 关键发现

1. **多流有效**: Mars X201通过2流配置提升6.4%端到端性能
2. **平台差异**: RTX 4090不受益于多流
3. **数据传输主导**: 77-83%的端到端时间是数据传输
4. **最优配置**: 2流（更多流没有额外收益）

### 6.2 推荐配置

**Mars X201:**
```cpp
int numStreams = 2;
int rowsPerStream = (numRows + numStreams - 1) / numStreams;
for (int s = 0; s < numStreams; s++) {
    int startRow = s * rowsPerStream;
    int endRow = min((s + 1) * rowsPerStream, numRows);
    spmv_range<<<gridSize, blockSize, 0, streams[s]>>>(
        startRow, endRow, ...);
}
```

**RTX 4090:**
```cpp
// 单流即可
spmv<<<gridSize, blockSize>>>(numRows, ...);
```
