## 2. 关键发现

### 2.1 Warp Size是决定因素

```
Mars X201 (Warp=64):
- avgNnz=10.71, 每行平均10.71个元素
- 如果1 warp处理1行: 理论利用率 = 10.71/64 = 16.7%
- 实际测量: 296.5 GB/s, 利用率 16.1%

TPR优化:
- TPR=8: 每warp处理8行, 每行8线程
- 每线程处理 ~10.71/8 = 1.34个元素
- 更多的并发线程 → 更好的延迟隐藏
- 带宽利用率从16%提升到48.7%
```

### 2.2 ILP优化效果分析

**Mars X201**: ILP几乎无效 (+0.2% ~ +0.5%)
- 原因: 内存带宽瓶颈，计算不是瓶颈
- 每线程只处理~10个元素，ILP无法发挥作用

**RTX 4090**: ILP反而降低性能 (-14% ~ -33%)
- 原因: L2 Cache足够大(72MB)，可以缓存x向量
- __ldg的纹理缓存访问比ILP更重要
- ILP增加了寄存器压力，减少了缓存效果

### 2.3 __ldg效果差异

**Mars X201**: __ldg无效 (+0.4%)
- 原因: L2 Cache太小(~2-4MB)，无法缓存10.8MB的x向量
- __ldg的纹理缓存无法弥补L2的不足

**RTX 4090**: __ldg有效 (+8.7%)
- 原因: L2 Cache足够大(72MB)，x向量(10.8MB)可以完全缓存
- __ldg提示让数据进入纹理缓存，加速后续访问

## 3. 理论分析

### 3.1 为什么TPR=8是Mars最优？

```
TPR与延迟隐藏的关系:

TPR=2: 每warp处理32行 (64/2=32)
- 每线程处理 ~5.3个元素
- 但每行只有8个活跃线程
- warp归约需要更多步

TPR=4: 每warp处理16行 (64/4=16)
- 每线程处理 ~2.7个元素
- 16个活跃线程，较好的并行度

TPR=8: 每warp处理8行 (64/8=8)
- 每线程处理 ~1.35个元素
- 8个活跃线程，最优延迟隐藏
- Warp内8组线程并行处理8行

TPR=16: 每warp处理4行 (64/16=4)
- 每线程处理 ~0.67个元素
- 只有4组，并行度降低
- 归约开销增加
```

### 3.2 为什么Mars利用率上限是49%？

```
理论计算:
- Mars X201 理论带宽: 1843 GB/s
- 实测带宽: 896.6 GB/s
- 利用率: 48.7%

可能的原因:
1. L2 Cache太小 (~2-4MB)
   - x向量大小: 10.8 MB
   - 无法缓存，每次访问都走DRAM

2. 随机访问模式
   - colIdx是随机的
   - 无法利用coalescing访问

3. Warp Size=64的限制
   - 更大的warp意味着更高的同步开销
   - 更难达到完全的延迟隐藏

4. 内存控制器限制
   - Mars的内存控制器可能不如NVIDIA优化
```

### 3.3 为什么RTX 4090达到91%利用率？

```
关键因素: L2 Cache (72MB)
- x向量大小: 10.8 MB
- 可以完全缓存在L2
- 第一次访问后，后续访问命中L2
- 带宽利用率可以超过100%（因为使用了缓存带宽）

__ldg的作用:
- 提示数据进入纹理缓存(read-only cache)
- 纹理缓存与L1/L2分离，提供额外的带宽
- 这解释了__ldg带来的+8.7%提升
```

## 4. 进一步优化方案

### 4.1 已尝试但无效的方案

| 方案 | Mars效果 | RTX效果 | 原因 |
|------|----------|----------|------|
| ILP优化 | +0.5% | -33% | 内存瓶颈 |
| 多流并行 | 无效 | 无效 | SpMV是内存受限 |
| 分批处理 | 负优化 | 负优化 | kernel启动开销 |
| CSR5格式 | -44% | - | 原子操作开销 |
| RCM重排序 | +1~15% | +3% | 真实矩阵缺乏结构 |

### 4.2 可能的新优化方案

1. **寄存器缓存部分x向量**
   - 如果某些列访问频繁，可以用寄存器缓存
   - 但随机访问模式下效果有限

2. **矩阵分块**
   - 将矩阵分成L2可缓存大小的块
   - 但会增加预处理开销

3. **列重排序**
   - 重排colIdx使得相邻访问更连续
   - 但需要预处理，且效果取决于矩阵结构

4. **混合精度**
   - FP32可以达到更高的带宽
   - 但精度降低

## 5. 最终结论

### 5.1 Mars X201最优配置

```cpp
// FP64 SpMV最优配置
int threadsPerRow = 8;   // TPR=8最优
int blockSize = 256;    // Block=256最优
int gridSize = (numRows * 8 + 255) / 256;

// Cache配置
cudaFuncSetCacheConfig(kernel, cudaFuncCachePreferL1);

// Pinned Memory (端到端优化)
cudaMallocHost(&h_x, numCols * sizeof(double));
```

### 5.2 RTX 4090最优配置

```cpp
// FP64 SpMV最优配置
int threadsPerRow = 1;   // 1 warp/row足够 (因为L2缓存)
int blockSize = 256;
int gridSize = (numRows * 32 + 255) / 256;

// 使用__ldg提示
// values[i] * __ldg(&x[colIdx[i]]) 获得更好效果

// Cache配置
cudaFuncSetCacheConfig(kernel, cudaFuncCachePreferL1);
```

### 5.3 性能差距分析

| 指标 | Mars X201 | RTX 4090 | 差距 |
|------|-----------|----------|------|
| 内核时间 | 0.421 ms | 0.409 ms | Mars慢3% |
| 带宽利用率 | 48.7% | 91.5% | Mars低2倍 |
| 理论带宽 | 1843 GB/s | 1008 GB/s | Mars高1.83倍 |
| **有效带宽** | 896 GB/s | 921 GB/s | **接近!** |

**关键发现**: 两台GPU的有效带宽相近（896 vs 921 GB/s），说明TPR=8优化后，Mars X201已经接近其硬件极限。

## 6. 端到端优化建议

### 6.1 Pinned Memory至关重要

之前的测试表明，Pinned Memory可以带来:
- Mars X201: +186% 端到端提升
- RTX 4090: +152% 端到端提升

### 6.2 综合优化流程

```cpp
// 1. 使用Pinned Memory
cudaMallocHost(&h_x, numCols * sizeof(double));
cudaMallocHost(&h_y, numRows * sizeof(double));

// 2. 根据GPU选择最优TPR
int warpSize = prop.warpSize;
int tpr = (warpSize == 64) ? 8 : 1;

// 3. 设置Cache配置
cudaFuncSetCacheConfig(kernel, cudaFuncCachePreferL1);

// 4. 执行kernel
spmv_kernel<<<gridSize, blockSize>>>(...);

// 5. NVIDIA额外使用__ldg
// if (warpSize == 32) use_ldg_hint();
```
