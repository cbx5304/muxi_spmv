# Mars X201 SpMV 深度性能分析报告

## 执行摘要

本报告记录了对Mars X201 GPU上稀疏矩阵向量乘法(SpMV)的深度性能分析，探索了矩阵形状对性能的影响。

## 关键发现：矩阵形状对性能的影响

### 列数影响（固定1M行）

| 列数 | NNZ | 带宽利用率 | 提升 |
|------|-----|-----------|------|
| 500 | 5M | 9.5% | 基准 |
| 1000 | 10M | 14.5% | +53% |
| 2000 | 20M | 23.9% | +152% |

**关键洞察**: 列数增加2倍，带宽利用率提升约1.5倍。

### 行数影响（固定2000列）

| 行数 | NNZ | 带宽利用率 |
|------|-----|-----------|
| 250K | 5M | 13.2% |
| 500K | 10M | 22.2% |
| 1M | 20M | 23.5% |

**关键洞察**: 行数对性能影响较小（当有足够列数时）。

## 性能瓶颈根因分析

### 1. x向量访问模式

```
性能 = f(列数)
```

**原因分析**:
- SpMV核心操作: `y[i] += A[i,j] * x[j]`
- x[j]的访问模式由colIdx决定，本质上是随机的
- 当列数增加时:
  - x向量变大，但访问密度降低
  - 缓存命中率可能提高（工作集更大，但访问更分散）
  - L2 cache可以缓存更多的x元素

### 2. rowPtr访问模式

```
rowPtr大小 = (numRows + 1) * sizeof(int)
```

| 行数 | rowPtr大小 | L2 Cache覆盖率 |
|------|-----------|---------------|
| 250K | 2 MB | ~50-100% |
| 500K | 4 MB | ~25-50% |
| 1M | 8 MB | ~12-25% |
| 2M | 16 MB | ~6-12% |

**L2 Cache限制**: Mars X201的L2 cache约2-4MB，当rowPtr超过此大小时，性能下降。

### 3. 数学分析

对于avgNnz=10的矩阵，每行处理需要:
- 读取rowPtr: 2次 * 8 bytes = 16 bytes
- 读取colIdx: 10 * 4 bytes = 40 bytes
- 读取values: 10 * 4 bytes = 40 bytes
- 读取x: 10 * 4 bytes = 40 bytes
- 写入y: 4 bytes

总计: 140 bytes per row

**理论带宽利用率**:
- 实测带宽: ~264 GB/s (14.3%利用率)
- 有效数据量: 140 bytes * 1M rows = 140 MB
- 理论最小时间: 140 MB / 1843 GB/s = 0.076 ms
- 实测时间: 0.33 ms
- 效率: 23%

**性能损失来源**:
1. 随机内存访问惩罚: ~2-3x
2. Merge-path搜索开销: ~10-20%
3. Kernel启动开销: ~5-10%
4. L2 cache miss: ~2x

## 与RTX 4090对比

### 硬件差异

| 参数 | Mars X201 | RTX 4090 | 差距 |
|------|-----------|----------|------|
| L2 Cache | ~2-4 MB | 72 MB | 18-36x |
| 带宽利用率 (avgNnz=10) | 14.3% | ~86% | 6x |
| 执行时间 (1M x 1K) | 0.33 ms | 0.037 ms | 9x |

### 根本差距原因

1. **L2 Cache大小**: RTX 4090的72MB可以缓存整个rowPtr + 部分x向量
2. **内存控制器**: NVIDIA的内存控制器处理随机访问更有效
3. **Warp效率**: RTX 4090的warp=32更容易充分利用

## 优化方向评估

### 已验证无效的优化

| 方案 | 结果 | 原因 |
|------|------|------|
| 共享内存缓存 | -33% | 加载开销 > 收益 |
| __ldg只读缓存 | 0% | 硬件不支持 |
| ELLPACK | +6%, +180%内存 | 行长度变化大时不适用 |
| CSR5 (原子操作版) | 8.7% | 原子操作开销过大 |
| Warp-cooperative merge | -80% | 同步开销过大 |

### 潜在优化方向

1. **列重排序**
   - 预处理: 根据访问模式重新排列列
   - 目标: 提高x向量的缓存局部性
   - 预期收益: 10-30%

2. **行分组处理**
   - 将行按NNZ数量分组
   - 大行用vector kernel，小行用scalar kernel
   - 预期收益: 5-15%

3. **软件预取**
   - 在处理当前行时预取下一行的数据
   - Mars X201可能不支持，需要验证

## 结论

### 当前最佳配置

| avgNnzPerRow | 推荐Kernel | 预期利用率 |
|--------------|-----------|-----------|
| < 32 | Merge-based | 14-24% |
| 32-64 | Vector | 35-52% |
| > 64 | Vector | 50%+ |

### 性能极限

Mars X201在稀疏矩阵上的带宽利用率受到以下硬件限制:
1. L2 cache大小: ~2-4MB (根本瓶颈)
2. 随机访问效率: 内存控制器设计
3. Warp大小: 64线程

**结论**: 在avgNnz=10场景下，14-24%的带宽利用率已接近该硬件的实际极限。进一步优化需要:
1. 硬件改进（增大L2 cache）
2. 矩阵预处理（列重排序）
3. 算法改进（减少随机访问）
